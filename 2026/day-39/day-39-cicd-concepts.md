### Task 2: CI vs CD
Research and write short definitions (2-3 lines each):
1. **Continuous Integration** — what happens, how often, what it catches
- When a developer commits code to the shared repository (e.g., Git), the following automated steps occur:
  ```
  Code Commit -> Automated Build -> Automated Testing -> Static Analysis & Linting -> Security Scanning -> Feedback Loop -> Artifact Creation
  ```
- CI should run every time a developer commits code to the shared repository.
- It catches: Integration Errors, Broken Builds, Test Regressions, Code Quality Issues, security vulnerabilities, and environment drifts.

2. **Continuous Delivery** — how it's different from CI, what "delivery" means
- Continuous Integration (CI): Focuses on the development phase. It automatically builds and runs unit tests whenever a developer commits code to the repository to detect bugs early.
- Continuous Delivery (CD): Picks up where CI leaves off. While CI ensures code is tested, CD ensures code is deployable. It automates the packaging and deployment to staging/production-like environments, enabling the team to release software at any moment.
- "Delivery" in CD means that every code change that passes the automated testing phase is automatically moved into a production-like environment (staging or pre-production). It is "delivered" to a stage where it is thoroughly tested and verified by automated processes and, potentially, QA teams, ensuring it is ready for release.
3. **Continuous Deployment** — how it differs from Delivery, when teams use it
- Human Approval (The Key Difference): Continuous Deployment requires no human intervention—changes go to production automatically. Continuous Delivery, however, requires a manual, "one-click" approval by a human to move changes from a staging environment to production.
- Teams adopt Continuous Deployment when they aim for maximum velocity, high-frequency, small-batch updates.

Write one real-world example for each.

### Task 3: Pipeline Anatomy
A pipeline has these parts — write what each one does:
- **Trigger** — what starts the pipeline
  - An event that kicks off the automated process, such as a Git push, a pull request, a scheduled time, or manual action.
- **Stage** — a logical phase (build, test, deploy)
  - A conceptual, sequential grouping of tasks (stages) that defines the order of execution (e.g., compile code, then run tests, then deploy).
- **Job** — a unit of work inside a stage
  - A set of commands executed within a stage; multiple jobs within the same stage can run in parallel.
- **Step** — a single command or action inside a job
  - The smallest, sequential building block within a job (e.g., running a script, installing dependencies, or echoing a command).
- **Runner** — the machine that executes the job
  - An agent or server (often a container or virtual machine) that picks up the job from the queue and executes the steps.
- **Artifact** — output produced by a job
  - Files, binaries, or logs generated by a successful job (e.g., compiled .jar file or Docker image) that can be passed to future stages or stored.
